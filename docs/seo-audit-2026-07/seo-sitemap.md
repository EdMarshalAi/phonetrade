# SEO-аудит: seo-sitemap — PhoneTrade

_Дата: 2026-07-02. Метод: skill `seo-sitemap` (валидация + quality-gates) + чтение `src/app/sitemap.ts` + живой sitemap.xml._

## Что проверяет скил
Валидность XML, лимит 50k URL, все URL → 200, точность `<lastmod>` (не одинаковые), отсутствие устаревших `<priority>/<changefreq>` (Google их игнорирует), ссылка на sitemap в robots.txt, отсутствие в карте noindex/redirect/non-canonical URL, HTTPS-only, разбивка по типам контента, quality-gates для программатик-страниц.

## Что у нас уже хорошо
- **Карта генерируется динамически** (`src/app/sitemap.ts`, ISR `revalidate=3600`): статик-роуты + категории + опубликованные товары + `static_pages` + посты блога. Новая страница/товар из админки попадает в карту в течение часа — без редеплоя.
- **Отдаётся 200, `content-type: application/xml`**, HTTPS-only URL. Проверено: **431 URL** — намного ниже лимита 50 000, разбивать не нужно.
- **Google Image extension**: у товаров `images: [p.image]` (`sitemap.ts:46`) — главное фото из Storage (абсолютный URL). Помогает попасть в Яндекс/Google Картинки по «iphone белгород».
- **`<lastmod>` реалистичные**: 154 различных значения из 431 URL (не «все одинаковые») — товары берут `p.updatedAt`, посты `published_at`. Скил специально флагует «all identical lastmod» — у нас этого нет.
- **Ссылка на sitemap в robots.txt** есть (`Sitemap: https://phonetrade31.ru/sitemap.xml`).
- **Нет мусорных разделов**: `/search`, `/cart`, `/account`, `/auth`, `/admin` в карту НЕ попадают (их нет в списках) и они закрыты в robots — согласовано, без конфликта «в карте, но disallow».

## Находки / проблемы (конкретно, файл/URL)

1. **Используются `priority` и `changeFrequency`** (`sitemap.ts:24-61`). По методологии скила это уровня **Info**: Google их **полностью игнорирует**, Яндекс — фактически тоже почти не учитывает. Не вредит, но и не помогает — «шум». `MetadataRoute.Sitemap` в Next их рендерит в `<priority>/<changefreq>`.

2. **`lastModified` статик-роутов и категорий = `now` (текущее время генерации)** (`sitemap.ts:24-38`). То есть у главной, `/catalog`, `/new`, `/used`, `/repair`, `/blog` и **всех категорий** дата меняется каждый час на «сейчас». Это ложный сигнал свежести: краулер видит, что раздел якобы обновляется ежечасно, хотя контент не менялся. Скил помечает неточные lastmod. У товаров/постов — правильно (реальный `updatedAt`), а вот категории берут `now` вместо `MAX(updatedAt)` товаров категории или даты правки самой категории.

3. **Нет разбивки по типам** (единый файл). При 431 URL это ОК (скил рекомендует разбивку только при росте и >50k), но на будущее — заметка.

4. **Проверка «все URL → 200» не автоматизирована.** Товары берутся из `getSitemapProducts()` — если товар архивирован/снят, но геттер его отдаёт, в карте окажется 404/редирект. Нужно убедиться, что геттер фильтрует только `is_published=true` и не-архив. (Требует проверки `src/lib/products.ts` — по коду `getSitemapProducts` отдаёт `image`, фильтр статуса не виден из sitemap.ts.)

5. **`/trade-in` в статик-роутах** (`sitemap.ts:28`) — это `static_pages` slug. Проверить, не дублируется ли он ещё раз через `pages.map` (`getPublishedPageSlugs`). Если `trade-in` опубликован как static_page, он попадёт в карту **дважды** (в staticRoutes и в pageRoutes) с разными priority — дубль URL в sitemap.

## Рекомендации

### 🔴 Критично
- (нет критичных — карта функционально здорова)

### 🟡 Важно
- **Убрать `now` из lastmod категорий/статик-разделов.** Для категории брать `MAX(updatedAt)` её товаров (или дату правки категории), для `/catalog`/`/new`/`/used` — максимум по релевантным товарам. Иначе теряется доверие к сигналу свежести и Яндекс может реже переобходить реально обновившиеся разделы.
- **Проверить и устранить возможный дубль `/trade-in`** (staticRoutes vs pageRoutes). Дедупить итоговый массив по `url` перед возвратом (`sitemap.ts:63`).
- **Гарантировать, что в карту не попадают снятые/архивные товары** — свериться, что `getSitemapProducts()` фильтрует `is_published` и исключает `type='used'`-архив/скрытые (`isIndexable=false`). Товар с `robots: noindex` (`product.isIndexable===false`, см. `product/[id]/page.tsx:29,34`) **не должен** быть в sitemap.

### 🟢 Nice-to-have
- Удалить `priority`/`changeFrequency` из sitemap (косметика — уменьшит размер, уберёт игнорируемые теги). Не срочно.
- При росте до ~2-3k+ URL — разбить на sitemap index (`sitemap-products`, `sitemap-blog`, `sitemap-pages`) через несколько `sitemap.ts` c `generateSitemaps()`.

## Быстрые победы (за час)
1. Дедуп итогового массива по `url` (одна строка) — снимает риск дубля `/trade-in`.
2. Аудит `getSitemapProducts()` на фильтр `is_published` + исключение `isIndexable=false` — чтобы noindex-товары не светились в карте (конфликт «в sitemap, но noindex» скил считает High).

## Google vs Яндекс (нюансы под наш кейс — топ Белгород)
- **`<image:image>` в sitemap** — Яндекс.Вебмастер и Google оба используют для Картинок; для магазина техники это канал трафика по «iphone 17 фото / купить белгород». У нас есть — плюс.
- **`priority`** Яндекс когда-то учитывал больше Google, но сейчас оба практически игнорируют — не полагаться.
- **Свежесть `lastmod`** для Яндекса важна при принятии решения о переобходе. Фиктивный `now` у категорий → Яндекс может «привыкнуть», что раздел меняется ежечасно, и либо чаще жечь краулинг-бюджет впустую, либо, наоборот, обесценить сигнал. Правильные lastmod помогают быстрее подхватывать реально изменившиеся цены/наличие — прямо в тему приоритета Яндекса.
- Карта + **IndexNow** (см. seo-technical) — связка для мгновенной переиндексации в Яндексе: sitemap для полного обхода, IndexNow для точечных апдейтов цен.
