# SEO-аудит: seo-drift — PhoneTrade

_Дата: 02.07.2026. Скил: seo-drift (Git для SEO: baseline → compare → history)._

## Что проверяет скил
Мониторинг «дрейфа» on-page SEO: скил снимает baseline (снимок) ключевых элементов страницы — `title`, meta description, canonical, meta robots, H1/H2/H3, JSON-LD, Open Graph, Core Web Vitals, HTTP-статус, хеши HTML и схемы — кладёт в локальную SQLite и при следующем прогоне сравнивает по **17 правилам в 3 уровнях серьёзности** (CRITICAL / WARNING / INFO). Отвечает ровно на наш вопрос после 12.06: **«что именно поменялось на страницах и когда»**. Это НЕ трекер позиций Яндекса — он ловит причину (изменения на странице), а не следствие (позиции). Позиции смотрим в Яндекс.Вебмастере параллельно.

## Что у нас уже хорошо
- **Стабильная генерация head-тегов из кода** (`generateMetadata` в каждом route) — снимок будет чистым и воспроизводимым, без случайного шума. Проверено на живом сайте: `/category/iphone` отдаёт корректный `<title>`, `meta description`, `<link rel="canonical" href="https://phonetrade31.ru/category/iphone">`.
- `html lang="ru"` присутствует, canonical самоссылочный — базовые поля, которые скил отслеживает, в норме.
- JSON-LD богатый и предсказуемый: `WebSite`+`SearchAction` (корень), `Store/ElectronicsStore` (site layout), `BreadcrumbList`+`ItemList` (категория), `Service` (repair), `Product/Offer` (товар). Скилу есть что хешировать (`schema_hash`) и отслеживать удаление (Rule 1) / модификацию (Rule 14).
- Robots.txt под Яндекс уже грамотный: `Clean-param` (color/memory/sort/model/sim/condition + utm/yclid/gclid), `Host`, `Sitemap` — это снижает риск того, что «дрейф» вызван размножением URL с параметрами.

## Находки / проблемы (конкретно)
1. **Дрейф после 12.06 никто не зафиксировал** — baseline до массового залива не снимался, поэтому *задним числом* восстановить «что именно поменялось в мете 30 просевших запросов» скилом нельзя. Скрипты скила требуют «известного хорошего» снимка ДО изменения. Это главный урок: baseline надо было снять перед заливом.
2. **Массовый залив 12.06 = массовый дрейф `title`/`meta_description`/`description_html`** у 317 товаров и категорий (документировано в CLAUDE.md: `seo-refine-products.ts`, `seed-category-seo-belgorod.ts`). По правилам скила это Rule 9 (Title Changed) + Rule 10 (Meta Changed) + Rule 17 (Content Hash) на сотнях URL одновременно — ровно тот профиль изменений, который Яндекс отвечает волатильностью и пере-индексацией. Память проекта (`yandex-rank-drop-2026-06-13`) это подтверждает.
3. **Дрейф контента правится из админки и seed-скриптов в обход review** — правки `meta_title`/`meta_description`/`seo_text` категорий и товаров прилетают через Supabase-мутации и `revalidatePath`, без снимка «до/после». Нет deployment-check-шага.
4. **CWV-часть скила у нас работать не будет без ключа PageSpeed** — `pagespeed_check.py` требует API-ключ; правила Rule 11/12 (регресс CWV) будут пропущены (`--skip-cwv`). Для нас это ок: CWV меряем через CrUX/Вебмастер, но помни при интерпретации отчёта.
5. **Дубль FAQPage** (из памяти `yandex-rank-drop`) — это как раз тип находки Rule 14 (Schema Modified) / валидности схемы, который скил бы поймал при сравнении.

### 🔴 Критично
- **Снять baseline ЭТАЛОННЫХ шаблонных URL прямо сейчас** и сделать это обязательным пред-деплойным шагом. Минимальный набор (10–12 URL, покрывают все типы шаблонов):
  `/`, `/category/iphone`, `/category/ipad`, `/category/mac`, один товар нового iPhone, один товар Б/У, `/new`, `/used`, `/repair`, `/trade-in`, одна geo-страница `/kupit-apple-staryj-oskol`, один пост блога.
  ```
  /seo drift baseline https://phonetrade31.ru/category/iphone
  ```
  После КАЖДОГО массового SEO-залива/seed-скрипта — `compare` по тем же URL и разбор CRITICAL/WARNING перед тем, как ждать реакции Яндекса.
- **Пред-деплойный гейт в CI.** В `.github/workflows/deploy.yml` после успешного деплоя гонять `drift compare` по эталонным URL и падать/предупреждать на CRITICAL (noindex добавлен, canonical сменился/пропал, title/H1 исчез, статус → 4xx/5xx). Это дешёвая страховка от повторения 12.06.

### 🟡 Важно
- **Завести историю намеренных изменений.** Каждый массовый seed-скрипт (`seo-refine-products.ts` и т.п.) должен логировать «до/после» в тот же `drift`-снимок, чтобы при просадке отличать «мы сами поменяли мету» от «Google/Яндекс что-то сделал».
- **Отдельный baseline для 30 просевших запросов.** Возьми URL, ранжировавшиеся по этим запросам (категории/товары), сними baseline и раз в неделю `compare` — увидишь, стабилизировалась ли мета или её продолжают дёргать из админки.
- **Мониторить дубль-схемы.** После фикса дубля FAQPage — baseline страницы с FAQ, чтобы повторный дубль ловился Rule 14.

### 🟢 Nice-to-have
- Хранить `drift/baselines.db` вне `~/.cache` (там она эфемерна для CI) — например, коммитить снапшоты эталонных URL в `docs/seo-baselines/` как JSON, чтобы история переживала машины.
- Раз в месяц `history` по эталонным URL — ретроспектива всех изменений.

## Google vs Яндекс
- **Яндекс важнее и он — причина текущей боли.** Яндекс сильнее реагирует на массовые одномоментные изменения меты/контента переиндексацией и временной волатильностью (что мы и видим с 12.06). Drift-мониторинг тут — способ доказать себе «мы больше НЕ трогаем мету пачками», дать индексу устаканиться и не паниковать раньше времени.
- **Google** переваривает изменения меты мягче и часто переписывает `title`/`description` сам — для него drift-снимок скорее подтверждает индексируемость (canonical/robots), чем предсказывает позиции.
- Скил **агностичен к поисковику** (он читает наш HTML, а не SERP), поэтому одинаково полезен для обоих. Для Яндекс-специфики (позиции, «выпавшие» запросы, региональность) он не замена Яндекс.Вебмастеру — используем в связке: Вебмастер показывает *что просело*, drift показывает *что мы поменяли*.
