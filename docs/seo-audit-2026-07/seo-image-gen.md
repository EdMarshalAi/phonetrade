## Что проверяет скил
Генерацию SEO-ассетов через AI: OG/social-превью (1200×630), hero-баннеры блога, schema-картинки (`ImageObject`), инфографику, favicon. Аудит-режим (spawned из `/seo audit`): анализирует существующие OG/social-картинки по сайту и выдаёт приоритизированный план генерации + пост-обработку (WebP, размер, alt, `ImageObject`/`og:image`-метатеги). Требует banana/nanobanana MCP; в проекте вместо него доступен **higgsfield MCP** (nano-banana) — им уже генерировали баннеры писем.

## Что у нас уже хорошо
- **Дефолтный OG для сайта — реальное фото магазина** (`layout.tsx` root: `OG_IMAGE` = `content/store-belgorod.jpg`, 1400×1400, с alt и гео). Для соцсетей (Telegram/ВК) живое фото магазина заходит лучше сгенерённого баннера — осознанное решение владельца, разумное.
- **OG/Twitter метатеги полны**: `og:title`, `og:description`, `og:image` (+ width/height/alt), `twitter:card=summary_large_image` — и в root, и на карточке товара (`product/[id]/page.tsx:35-36`).
- **Баннеры email** сгенерированы под Apple-эстетику (nano-banana) + наложены лого/заголовок — качественная работа, единый стиль.
- **Шрифт для OG есть** — `src/app/_og/PTSans.ttf` (SIL OFL, коммерческое использование разрешено) — задел под динамический баннер.
- **favicon** присутствует (`favicon` в проекте), лого-марка `brand/logo-mark-black.png` используется как logo/image в Organization.

## Находки / проблемы (конкретно, файл/URL)

### 🔴 Динамический OG-баннер удалён — `opengraph-image.tsx` возвращает 404
`GET https://phonetrade31.ru/opengraph-image` → **404**. Файлов `src/app/opengraph-image.tsx`/`twitter-image.tsx` в репозитории НЕТ (`find src/app -name 'opengraph-image*'` пусто), хотя `CLAUDE.md`/`README` утверждают, что он есть (фирменный баннер 1200×630 через `next/og`, шрифт PT Sans). **Документация устарела — фичи в проде нет.** Остался только осиротевший `_og/PTSans.ttf`. Итог: OG-превью всех страниц (кроме товара) = статичное фото магазина 1400×1400 (квадрат, не 1200×630).

### 🟡 OG-картинка товара = «голое» фото товара на белом/прозрачном фоне
`product/[id]/page.tsx:35`: `openGraph.images = [productImages(product)[0]]`, т.е. каталожное фото товара. В ленте Telegram/ВК/поиске это выглядит как вырезанный предмет без контекста, без цены, без бренда PhoneTrade — слабый CTR при шеринге карточки товара. Скил рекомендует для product-OG: чистый кадр + подпись/цена/лого (domain mode Product, 16:9). Это лучший кандидат под динамический `next/og` баннер: фото товара + название + цена наличными + «PhoneTrade · Белгород».

### 🟡 Соотношение сторон OG не 1200×630
Дефолтный OG 1400×1400 (квадрат). Соцсети и поисковые превью рассчитаны на 1.91:1 (1200×630) — квадрат кропается непредсказуемо. Скил-пресет `og-default`: `magick ... -resize 1200x630^ -extent 1200x630`.

### 🟢 Нет `ImageObject`-схемы для ключевых баннеров
Скил предлагает `ImageObject` для сгенерированных ассетов. Для товарных фото избыточно (они уже в Product.image), но для hero/блога — nice-to-have.

### 🟢 banana MCP в проекте не сконфигурирован
Скил рассчитан на nanobanana MCP; в `~/.claude.json` его нет. Но доступен **higgsfield MCP** (nano-banana, тот же движок, которым делали баннеры писем) — генерацию гнать через него. Просто фиксирую, что «родной» инструмент скила не подключён.

## Google vs Яндекс (нюансы)
- OG/Twitter-теги — это про **шеринг в соцсетях** (Telegram/ВК/WhatsApp — основные каналы в РФ) и превью-карточки, а НЕ прямой фактор ранжирования ни в Google, ни в Яндексе. Приоритет по бизнес-эффекту: красивое товарное превью при пересылке ссылки в мессенджере даёт клики.
- **Яндекс** дополнительно использует og:image/og:title в собственных превью и в Дзене/Нейро — фирменный баннер узнаваем.
- **AI-лейбл**: если товарные OG будут AI-генерённые и попадут в фид — нужен IPTC `TrainedAlgorithmicMedia` (Merchant Center); для чистого OG/шеринга — не требуется. Сейчас product-OG = реальные фото, риска нет.
- Генерацию вести через доступный **higgsfield MCP** (banana/nanobanana в проекте не подключён), сохраняя серо-чёрную Apple-палитру проекта.
