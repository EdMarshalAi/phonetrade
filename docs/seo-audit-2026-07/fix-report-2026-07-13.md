# PhoneTrade: подтверждение SEO/commerce-исправлений — 13.07.2026

Статус: исправления реализованы локально и проверены на production-сборке. Яндекс.Вебмастер и данные в нём не изменялись. Ручных правок на production-сервере не было; доставка изменений — только `main` → GitHub Actions.

## Что было подтверждено до исправления

- 27 доступных архивных карточек отвечали прямым `200`, хотя товар уже не опубликован.
- `/used` и `/category/iphone-used` показывали одну коллекцию из 102 товаров.
- Три опубликованные категории были фактически пустыми, но могли попадать в sitemap и индекс.
- 165 из 301 товарных meta-description содержали старую цену; цена меняется автоматическим пересчётом.
- 84 новых товара имели числовой `stock=0`, но старые флаги сообщали «в наличии».
- YML создавал фиктивный `oldprice` из карточной цены; у 193 предложений это не было реальной старой ценой.
- В 43 описаниях не-Apple товаров встречалось утверждение «оригинал Apple»; у 15 Dyson отсутствовал бренд, Hollyland определялся как `Other`.
- В Product Offer публиковались неподтверждённые срок цены, доставка и возврат.
- `/account`, `/cart` и `/auth/login` не имели явной закрывающей index-policy.
- Пересчёт цены обновлял `products.updated_at`, из-за чего sitemap создавал ложную свежесть товаров.
- `published_at` статьи перезаписывался при обычном редактировании.
- В checkout клиент и сервер по-разному считали доставку; сервер доверял переданным ценам. При активном пересчёте итог мог измениться после подтверждения покупателем.
- Каталог сериализовал в браузер тяжёлые SEO-описания, характеристики и служебные поля, не нужные карточкам.

## Что исправлено

### Индексация и crawl

- Неопубликованный/удалённый товар больше не выдаётся через публичный product route.
- `/category/iphone-used` постоянно перенаправляется на `/used`; внутренние ссылки и Breadcrumb JSON-LD используют единственный canonical URL.
- Пустая категория получает `noindex,follow` и исключается из sitemap только при надёжном полном счётчике. Частичный ответ БД не меняет индексную политику.
- Возвращён `/trade-in`, удалён ложный товарный `lastmod`, даты статей разделены на публикацию и изменение.
- Приватные страницы получили `noindex,nofollow`.
- Robots очищен от устаревшего `Host`; подтверждённые facet/tracking/legacy-параметры находятся в группе Yandex.

### Цена, наличие и schema

- Текущая `price_cash` подставляется во все сохранённые форматы цены (`₽`, `руб.`, обычный/NBSP/narrow-NBSP/HTML entity) во время SSR.
- Получение товара обёрнуто в request-level `React.cache`: metadata и видимая страница используют один снимок цены даже во время пересчёта.
- Числовой `stock` стал источником истины. Разрешённый нулевой остаток — `BackOrder`; физического наличия нет, YML отдаёт `available=false`.
- `oldprice` выводится только из настоящего `price_old > price_cash`.
- Удалены неподтверждённые `priceValidUntil`, `shippingDetails`, `hasMerchantReturnPolicy`.
- Бренд не становится Apple по умолчанию; Dyson/Hollyland определяются явно, ложная Apple-фраза для других брендов заменяется нейтральной.

### Нагрузка и checkout

- Каталог, избранное, related и корзина выбирают только необходимые карточке поля вместо `select(*)` и тяжёлых HTML/specs.
- Повторные чтения категорий/настройки наличия/товара дедуплицируются внутри одного SSR-запроса.
- При настроенном Supabase ошибка БД больше не подменяется старыми mock-ценами; production работает fail-closed.
- Server Action ограничен 50 уникальными строками и 10 единицами позиции, повторно читает актуальные цены/наличие/оплату/доставку и не принимает денежные поля от браузера.
- Клиент передаёт только контрольный ожидаемый итог. Если во время пересчёта цена или доставка изменилась, заказ не создаётся: корзина обновляется, покупатель подтверждает новую сумму повторно.
- Заголовок заказа, строки, финальная проверка промокода, его счётчик и журнал применения записываются одной DB-транзакцией. Ошибка любой части откатывает всё; статистика клиента меняется только после успешного заказа.
- Товарные строки перечитываются и блокируются внутри этой же транзакции. Подтверждаются актуальные `price_cash`/`price_card`, subtotal, скидка наличных и итоговая арифметика; в режиме запрета продажи под заказ остаток проверяется и уменьшается под row-lock, поэтому параллельные заказы не создают oversell.
- Бесплатная доставка расходует промокод только при реальной выгоде. Строка промокода блокируется на время транзакции, поэтому общий, клиентский и `only_new_customers` лимиты нельзя обойти двумя параллельными заказами.
- Браузер хранит стабильный UUID одной попытки checkout до подтверждённого ответа. RPC и Server Action сначала восстанавливают уже зафиксированный заказ по этому ключу, поэтому timeout, reload или изменение цены после потерянного ответа не создают дубль и не повторяют stock/customer/promo counters.
- При обычной смене цены валидный промокод сохраняется, корзина и server props обновляются. При изменении/исчерпании самого промокода он снимается явно. Количество в UI и Server Actions ограничено фактическим остатком, когда продажа под заказ отключена.
- Комментарий, желаемое время и реквизиты юрлица сохраняются в заметке заказа.
- Telegram и письмо запускаются через поддерживаемый Next.js `after()` после HTTP-ответа; PM2 получает 30 секунд на корректное завершение callbacks при reload. На idempotent replay уведомления повторно не отправляются.

## Автоматические проверки

- `npx tsc --noEmit` — успешно.
- `npm run lint:critical` — успешно.
- `npm run test:seo` — 7/7.
- `npm run build` — production standalone собран успешно.
- Локальный production smoke — успешно: 6 эталонных URL, permanent redirect, архивный 404, 4 private noindex, robots, sitemap, 48 indexable-категорий, три товарных metadata/schema и YML из 199 offers.
- GitHub Actions валидирует обязательные secrets, запускает critical lint, typecheck, unit-regression и блокирующий smoke. После SQL проверяется видимость RPC через PostgREST без записи данных. Бандл сначала работает отдельным candidate на `127.0.0.1:3001` и проверяется через SSH tunnel; production-процесс на `:3000` переключается только после этого. Затем выполняется live smoke, атомарно меняется ссылка и только тогда сохраняется PM2. Ошибка/отмена откатывает предыдущий release, а watchdog чистит candidate при потере runner.

Полный `npm run lint` всё ещё показывает накопленный legacy-долг в старой админке и одноразовых import-скриптах (до этого патча). Эти несвязанные ошибки не маскируются: критичный storefront/SEO/checkout-контур выделен в обязательный чистый lint-гейт.

## Что сознательно не делали

- Не меняли Яндекс.Вебмастер, IndexNow-настройки или данные каталога вручную.
- Не запускали массовую генерацию/перезапись SEO-контента: предыдущий аудит предписывает паузу до 14.07 и последующие партии по 3–5 материалов.
- Не добавляли отзывы или `aggregateRating` без реальных источников.
- Не обещаем мгновенный выход в топ: релиз устраняет технические противоречия, но переобход и переоценка Яндексом занимают время.
