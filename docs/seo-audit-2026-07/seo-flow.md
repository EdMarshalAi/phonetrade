# SEO-аудит: seo-flow — PhoneTrade

> Дата: 2026-07-02. Скил `seo-flow` v2.2.0 — FLOW framework (Find → Leverage → Optimize → Win).
> Framework and prompts © Daniel Agrici, CC BY 4.0 — github.com/AgriciDaniel/flow

## Что проверяет/делает скил

`seo-flow` — не аудитор, а **операционный цикл** (рабочий процесс). Даёт 41 evidence-led промпт, разбитый на 5 стадий:

| Стадия | Смысл | Промптов |
|---|---|---|
| **Find** | спрос: keyword research, gap-анализ, SERP-intent | 5 |
| **Leverage** | внешний авторитет: беклинки, off-site evidence | 1 |
| **Optimize** | владельческие ассеты: extraction + trust (on-page, тех, контент) | 21 (выбираем 2–3) |
| **Win** | конверсия, BOFU, dual-surface scorecard | 3 |
| **Local** | GBP, мета, тайтлы, локальные аудиты | 11 |

Ключевой принцип FLOW-2026: рейтинги, AI-цитаты, локал-пак и продажи — это **связанные поверхности**, а не отдельные каналы. Правило: назвать поверхность до написания текста (органика / AI-ответ / локал-пак / карточка бизнеса / посадочная), отделять факты от допущений, писать сначала на языке покупателя. Диагностика «какая стадия блокирует прогресс»: спрос неясен → Find; бренд не подтверждён off-site → Leverage; ассет плохо извлекается/недоверчив → Optimize; трафик есть, продаж нет → Win.

## Как применить к нам

FLOW — идеальный **регулярный пайплайн** для PhoneTrade, потому что мы уже в фазе «трафик был → просел → надо вернуть и добрать». Прогонять не весь набор из 41 промпта, а стадию, которая блокирует прямо сейчас.

**Где мы застряли по FLOW-диагностике:**
- Спрос (Find) — известен: семантика собрана (`seo-keywords-*.csv`), кластеры есть. Find включаем ТОЧЕЧНО под 19 «невидимых» запросов.
- Leverage — **слабое звено**: беклинков почти нет, off-site держится на картах/каталогах. Для локального Apple-магазина это ок, но авторитет домена низкий.
- Optimize — техника закрыта (C1–C8), но есть регрессии после залива (мета не-iPhone, возможный дубль FAQPage).
- Win — не измеряется: нет связи «органика → заявка/звонок/заказ». Есть `page_views` с фильтром ботов, но воронка «поисковый визит → покупка» не выстроена.
- Local — сильная сторона (NAP, JSON-LD, карты), но **отзывов мало** — а это top-фактор и для локал-пака, и для AI-рекомендаций.

## Рекомендованный регулярный пайплайн FLOW для PhoneTrade

### Еженедельно (лёгкий цикл)
1. **Win → measurement:** снять позиции (Topvisor, регион Белгород), сверить с прошлой неделей. Отследить отыгрыш просадки. Отметить, какие гэп-статьи (от 25.06) начали ранжироваться.
2. **Local:** попросить 1–2 новых отзыва на Яндекс.Картах/2ГИС (QR на чеке). Проверить, что часы работы/NAP актуальны (часы — top-5 фактор локал-пака по Whitespark 2026).

### Ежемесячно (полный цикл)
3. **Find** (5 промптов) — только под новые модели/сезон и под оставшиеся «невидимые» запросы. НЕ пересобирать всю семантику (это триггерит массовый залив → сброс переоценки Яндекса).
4. **Optimize** (выбрать 2–3 из 21): для нас приоритет — `on-page + content` промпты (карточки, категории) и `technical` (проверка регрессий). Дампить все 21 — шум.
5. **Win** (3 промпта): выстроить dual-surface scorecard — органика Яндекс + AI-цитаты (Алиса/ChatGPT) → заявка/заказ. Добавить событие измерения там, где его нет.

### Ежеквартально
6. **Leverage** (1 промпт) — стратегия наращивания внешнего авторитета: локальные каталоги, отраслевые площадки, упоминания. **Без покупных прогонов** (для локального магазина риск фильтра выше пользы — зафиксировано в мастер-плане).

## Рекомендации

### 🔴 Критично
- **Win-стадия не измеряется** — нет воронки «поисковый визит → заказ». Пока не связали discovery с revenue, любой рост позиций оценивается только по impressions. Добавить событие «заказ с органики» в аналитику.

### 🟡 Важно
- **Leverage — самое слабое место.** Off-site держится на 2 картах. Нужен план внешних упоминаний (локальные СМИ Белгорода, каталоги, отраслевые площадки) — органически, не прогонами.
- **Local: отзывы.** Их мало. Для FLOW это одновременно Local (локал-пак) и Leverage (external evidence для AI). Самый дешёвый рычаг с высоким ROI.

### 🟢 Nice-to-have
- Прогонять Optimize-промпты перед каждым большим апдейтом контента как чеклист «извлекаемо ли это ИИ».
- dual-surface scorecard оформить в один дашборд (позиции Яндекс + AI-цитаты + заказы).

## Быстрые победы
1. Прогнать `/seo flow local` по главной — быстрый чеклист мета/тайтлов (тут же вылезет проблема домашнего `<title>` без интента).
2. Прогнать `/seo flow find` только по 19 «невидимым» запросам — выяснить точный intent, дописать мету поштучно.
3. Настроить один measurement-event «заказ из органического поиска» — без него Win-стадия слепая.

## Google vs Яндекс
- FLOW агностичен к движку, но **Leverage и Local важнее для Яндекса** (Яндекс сильнее опирается на поведенческие + карты/отзывы). Для Google важнее Optimize (техника, схема, CWV).
- **AI-поверхности:** Яндекс → Алиса/Нейро (тянет из Яндекс.Карт и Бизнеса — усиливается через Local); Google/западные ИИ → ChatGPT/Perplexity (тянут из Bing-индекса + внешних упоминаний — усиливается через Leverage). Обе стадии у нас недокачаны.
- Приоритет владельца — Яндекс, поэтому в FLOW-цикле вес: **Local > Win > Find > Optimize > Leverage** (Leverage долгий, но не бросать).
