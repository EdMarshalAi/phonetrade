# Промт для Claude Code — Модуль «Прайс» для PhoneTrade

> Скопируй этот файл в проект (`docs/pricing-module.md`) и используй как промт.
> Задача — добавить массовый редактор цен, привязанный к курсу USD, и сделать так чтобы цены автоматически тянулись из прайса в карточку товара.

---

## 0. ЦЕЛЬ

В админке PhoneTrade нужно сделать **единый источник цен** — раздел «Прайс», где цены рассчитываются автоматически из закупочной цены и курса USD по настраиваемой формуле. Карточка товара и публичный сайт ссылаются на эти рассчитанные цены, не дублируя логику.

**Принципиальная модель:**
- В товаре менеджер вводит **закупочную цену в ₽ + курс на момент закупа** (как привычно — поставщик выставляет в ₽)
- Из этого система автоматически вычисляет `cost_usd` (закупочная в долларах, фиксированная для товара)
- При публикации товара / при изменении курса USD в админке / при изменении настроек формулы — все цены товара (нал, карта, кредит 6/12/24 мес) **пересчитываются единой формулой** и сохраняются в БД
- Карточка товара, корзина, чекаут, заказы, аналитика — все читают уже готовые `price_cash`, `price_card` и т.д. из таблицы товаров. Никто не считает на лету.
- Менеджер может **зафиксировать цену вручную** (для акций / особых позиций) — флаг в товаре, формула отключается для этого SKU
- В разделе «Прайс» — массовое редактирование закупочных цен, изменение курса, изменение формулы, импорт/экспорт XLSX/CSV

## 1. ИСТОЧНИК КУРСА ВАЛЮТ

**Используем официальное API ЦБ РФ через зеркало:**
- Основное: `https://www.cbr-xml-daily.ru/daily_json.js` (JSON, бесплатный, без ключа, обновляется ежедневно по данным ЦБ)
- Запасное (если основное недоступно): `https://www.cbr.ru/scripts/XML_daily.asp` (официальный XML ЦБ)
- НЕ парсить investing.com — Terms of Use не разрешают, плюс это SPA с JS-рендерингом, нестабильно

**Структура ответа cbr-xml-daily:**
```json
{
  "Date": "2026-05-30T11:30:00+03:00",
  "PreviousDate": "2026-05-29T11:30:00+03:00",
  "Valute": {
    "USD": { "Value": 89.45, "Previous": 89.12, "Nominal": 1 },
    "EUR": { "Value": 96.78, "Previous": 96.55, "Nominal": 1 }
  }
}
```

**Что делаем:**
1. Создаём Edge Function / cron-задание Supabase, которое раз в час дёргает API ЦБ и пишет в таблицу `currency_rates` (USD, EUR, дата, источник). Если API недоступно — пишем в `notification_log` и шлём алёрт в Telegram.
2. В админке отдельная страница `/admin/settings/currency` со списком истории курсов (последние 30 дней) + кнопкой «Обновить вручную».
3. **Курс в формуле прайса — отдельное поле**, потому что менеджер часто хочет курс «ЦБ + 1.5%» (поправка на конвертацию у эквайринга). В прайсе это поле `working_usd_rate` (рабочий курс), а курс ЦБ показывается рядом как референс.

## 2. СХЕМА БД (миграции Supabase)

### Новые таблицы

```sql
-- История курсов ЦБ (для аналитики и отката)
currency_rates (
  id bigserial pk,
  date date not null,
  usd numeric(10,4) not null,
  eur numeric(10,4) not null,
  source text default 'cbr-xml-daily',
  fetched_at timestamptz default now(),
  unique (date, source)
)

-- Настройки прайса (одна запись с id=1)
pricing_settings (
  id int pk default 1,
  -- Курсы
  working_usd_rate numeric(10,4) not null,  -- рабочий курс USD, который менеджер задаёт
  use_cbr_auto bool default false,           -- автоматически тянуть курс ЦБ
  cbr_markup_percent numeric(5,2) default 0, -- наценка на курс ЦБ (например 1.5%)
  -- Наценки
  fx_markup_percent numeric(5,2) default 10,       -- наценка на курс (то что в Excel +10%)
  card_markup_percent numeric(5,2) default 15,     -- карта над налом
  credit_6m_markup_percent numeric(5,2) default 23,
  credit_12m_markup_percent numeric(5,2) default 28,
  credit_24m_markup_percent numeric(5,2) default 37,
  -- Округление
  price_rounding int default 1000,                  -- округлять до тысяч
  -- Защита
  min_margin_percent numeric(5,2) default 5,        -- минимальная маржа над закупом
  -- Метаданные
  updated_at timestamptz default now(),
  updated_by uuid fk → admin_users.id
)

-- История изменений цен товаров (для аудита)
product_price_history (
  id bigserial pk,
  product_id uuid fk → products.id,
  cost_rub numeric,
  cost_rate numeric,
  cost_usd numeric,
  price_cash numeric,
  price_card numeric,
  credit_6m_total numeric,
  credit_12m_total numeric,
  credit_24m_total numeric,
  reason text,           -- 'manual_edit' | 'fx_recalc' | 'formula_change' | 'import'
  changed_by uuid fk → admin_users.id null,
  changed_at timestamptz default now()
)
```

### Изменения в таблице products

```sql
alter table products add column cost_rub numeric;
alter table products add column cost_rate numeric;        -- курс на момент закупа
alter table products add column cost_usd numeric generated always as (cost_rub / nullif(cost_rate, 0)) stored;
alter table products add column price_override bool default false;  -- ручная блокировка цены
alter table products add column credit_6m_total numeric;            -- результат формулы Q→J
alter table products add column credit_6m_monthly numeric;          -- J/6
alter table products add column credit_12m_total numeric;
alter table products add column credit_12m_monthly numeric;
alter table products add column credit_24m_total numeric;
alter table products add column credit_24m_monthly numeric;
alter table products add column prices_recalculated_at timestamptz;
```

**Связь с прайсом:** прайс — это **витрина (view/query)** над таблицей products плюс расчётные колонки. Отдельной таблицы для прайса нет, иначе данные расходятся. Идентификация в массовом редактировании — по `id` товара (uuid), `sku` показываем как удобную для человека строку.

### RLS

Все три новые таблицы — RLS включён, доступ только для ролей `owner`, `admin`, `manager` (для пересчёта). `analytics` и `content` — только read у `currency_rates` (для информации).

## 3. ЛОГИКА ПЕРЕСЧЁТА ЦЕН

### Утилита `src/lib/pricing/calculate.ts`

```ts
type PricingSettings = {
  working_usd_rate: number;
  fx_markup_percent: number;
  card_markup_percent: number;
  credit_6m_markup_percent: number;
  credit_12m_markup_percent: number;
  credit_24m_markup_percent: number;
  price_rounding: number;
};

type ProductPriceInputs = {
  cost_usd: number;                  // вычисляется из cost_rub / cost_rate
  price_override: boolean;
  override_price_cash?: number;      // если price_override = true
  override_price_card?: number;
};

type CalculatedPrices = {
  price_cash: number;
  price_card: number;
  credit_6m_total: number;
  credit_6m_monthly: number;
  credit_12m_total: number;
  credit_12m_monthly: number;
  credit_24m_total: number;
  credit_24m_monthly: number;
  // диагностика
  base_cash: number;                 // до округления (внутреннее, для отладки)
};

function calculatePrices(
  product: ProductPriceInputs,
  settings: PricingSettings
): CalculatedPrices {
  // Если зафиксирована вручную — используем override
  if (product.price_override && product.override_price_cash) {
    const cash = product.override_price_cash;
    const card = product.override_price_card ?? round(cash * 1.15, settings.price_rounding);
    // Кредитные всё равно считаем по формуле, от зафиксированной цены нал
    return buildFromBaseCash(cash, card, settings);
  }
  
  // Стандартный путь: формула
  const workingRate = settings.working_usd_rate * (1 + settings.fx_markup_percent / 100);
  const baseCash = product.cost_usd * workingRate;
  
  const price_cash = round(baseCash, settings.price_rounding);
  const price_card = round(baseCash * (1 + settings.card_markup_percent / 100), settings.price_rounding);
  const credit_6m_total = round(baseCash * (1 + settings.credit_6m_markup_percent / 100), settings.price_rounding);
  const credit_12m_total = round(baseCash * (1 + settings.credit_12m_markup_percent / 100), settings.price_rounding);
  const credit_24m_total = round(baseCash * (1 + settings.credit_24m_markup_percent / 100), settings.price_rounding);
  
  return {
    price_cash, price_card,
    credit_6m_total, credit_6m_monthly: Math.round(credit_6m_total / 6),
    credit_12m_total, credit_12m_monthly: Math.round(credit_12m_total / 12),
    credit_24m_total, credit_24m_monthly: Math.round(credit_24m_total / 24),
    base_cash: baseCash,
  };
}

const round = (n: number, step: number) => Math.round(n / step) * step;
```

**КЛЮЧЕВОЕ:** все наценки считаются от `baseCash` (не округлённой), а не каскадно от `price_cash`. Это устраняет ошибку из исходного Excel где T = C × 1.15, а C уже округлено.

### Триггеры пересчёта

Цены пересчитываются в трёх случаях, **только server-side**:

1. **Создание / редактирование товара** — server action `upsertProduct()`: после сохранения `cost_rub` и `cost_rate` сразу вычисляет все цены и сохраняет в той же транзакции. Запись в `product_price_history` с `reason='manual_edit'`.

2. **Изменение настроек прайса / курса** — server action `updatePricingSettings()`: после сохранения настроек запускает фоновую пересборку всех товаров без `price_override`. Делается батчами по 100 товаров через `for ... loop` в Postgres-функции, чтобы не блокировать. Запись истории с `reason='formula_change'` или `'fx_recalc'`.

3. **Массовая операция в редакторе прайса** — соответствующий reason.

### Postgres-функция для массового пересчёта

```sql
create or replace function recalculate_all_prices(
  p_reason text default 'fx_recalc',
  p_user_id uuid default null
)
returns int
language plpgsql
security definer
as $$
declare
  v_settings record;
  v_product record;
  v_base_cash numeric;
  v_count int := 0;
begin
  select * into v_settings from pricing_settings where id = 1;
  
  for v_product in 
    select id, cost_usd, price_override, price_cash as old_cash
    from products 
    where price_override = false and cost_usd is not null
  loop
    v_base_cash := v_product.cost_usd * v_settings.working_usd_rate * (1 + v_settings.fx_markup_percent / 100.0);
    
    update products set
      price_cash = round(v_base_cash / v_settings.price_rounding) * v_settings.price_rounding,
      price_card = round(v_base_cash * (1 + v_settings.card_markup_percent / 100.0) / v_settings.price_rounding) * v_settings.price_rounding,
      credit_6m_total = round(v_base_cash * (1 + v_settings.credit_6m_markup_percent / 100.0) / v_settings.price_rounding) * v_settings.price_rounding,
      credit_12m_total = round(v_base_cash * (1 + v_settings.credit_12m_markup_percent / 100.0) / v_settings.price_rounding) * v_settings.price_rounding,
      credit_24m_total = round(v_base_cash * (1 + v_settings.credit_24m_markup_percent / 100.0) / v_settings.price_rounding) * v_settings.price_rounding,
      credit_6m_monthly = round(credit_6m_total / 6.0),
      credit_12m_monthly = round(credit_12m_total / 12.0),
      credit_24m_monthly = round(credit_24m_total / 24.0),
      prices_recalculated_at = now()
    where id = v_product.id;
    
    -- В историю
    insert into product_price_history (product_id, price_cash, price_card, reason, changed_by)
    select id, price_cash, price_card, p_reason, p_user_id from products where id = v_product.id;
    
    v_count := v_count + 1;
  end loop;
  
  return v_count;
end;
$$;
```

Вызов из server action: `await supabase.rpc('recalculate_all_prices', { p_reason: 'formula_change', p_user_id: userId })`.

### Инвалидация кэша

После любого пересчёта — `revalidateTag('products')` чтобы публичный сайт получил новые цены без полной пересборки.

В существующей карточке товара (`/admin/catalog/products/[id]`):

### Вкладка «Цены и наличие» — переработать

**Блок «Закупка»:**
- Поле «Закупочная цена, ₽» (`cost_rub`)
- Поле «Курс USD на момент закупа» (`cost_rate`) — с подсказкой «Курс по которому платили поставщику. Если не знаете — ставьте текущий курс ЦБ»
- Кнопка «Вставить текущий курс ЦБ» рядом — подставляет курс ЦБ на сегодня
- **Под полями вычисленное значение** мелким серым: «Закупочная цена в долларах: $832,90» (`cost_usd`)
- Подсказка-иконка: «Закупочная цена в долларах фиксируется при сохранении. Дальше все цены пересчитываются от этого значения и текущего курса USD в настройках прайса»

**Блок «Цены продажи» (read-only, с пометкой «Рассчитано автоматически»):**
- Цена нал — большая, красная
- Цена картой
- Рассрочка 6 мес / 12 мес / 24 мес — сумма и ежемесячный платёж
- Внизу строка «Маржа: X% (X ₽)» — разница между ценой нал и закупкой в текущих ценах. Если ниже `min_margin_percent` — красный алёрт
- Под блоком: «Цены пересчитаны: 30 мая 2026, 14:23» + кнопка «Обновить из формулы»

**Toggle «Зафиксировать цену вручную»** (`price_override`):
- По умолчанию выключен — работает формула
- При включении блок «Цены продажи» становится редактируемым (только нал и карта), кредитные пересчитываются от введённой цены нал
- Появляется предупреждение: «Цена не будет пересчитываться при изменении курса. Не забудьте снять флажок после акции»
- Над ценой — бейдж «Зафиксировано вручную»

**Блок «История цен»** (свёрнутый по умолчанию):
- Таблица из `product_price_history` для этого товара
- Колонки: дата, цена нал, цена карта, причина изменения, кто изменил

### Шапка с курсами

**Большие плитки в шапке (sticky при скролле):**
- **USD ЦБ** — текущий курс ЦБ, дельта к вчерашнему (стрелка зелёная/красная), время последнего обновления
- **EUR ЦБ** — то же самое (для справки, в формуле не используется)
- **Рабочий курс USD** — то что менеджер задаёт в формуле. Inline-edit. Рядом мини-формула: «89.45 (ЦБ) × 1.5% = 90.79». Кнопка «Подставить из ЦБ» с дропдауном «+0%», «+1%», «+1.5%», «+2%», «+3%»

Если `use_cbr_auto = true` в настройках — рабочий курс **read-only**, обновляется автоматически каждый час. Бейдж «🔄 Автообновление».

### Кнопки действий в шапке

- **⚙️ Формула** — открывает drawer с настройками `pricing_settings`:
  - Все наценки (FX, карта, кредиты) с инлайн-превью «При курсе X и закупке $1000 цена нал будет = Y ₽»
  - Округление
  - Минимальная маржа (для алёртов)
  - Toggle автообновления курса ЦБ + размер поправки
  - **Кнопка «Сохранить и пересчитать»** — применяет формулу ко всем товарам без `price_override`. Показывает progress bar: «Пересчитано 247 из 891 товаров...». В конце тост «Пересчитано 891 товар. Изменилось 643».

- **📥 Импорт** — открывает модалку «Импорт прайса»:
  - Drag-and-drop поле для XLSX/CSV
  - Селектор «Идентификатор товара»: SKU / ID (по умолчанию SKU)
  - Селектор «Что обновлять»: только закупку / только финальные цены (с автовключением `price_override`) / и то и то
  - Парсит файл, показывает **предпросмотр изменений**: «Будет обновлено 423 товара, 12 не найдены, 5 без изменений»
  - Таблица предпросмотра с подсветкой изменений (старое → новое)
  - Кнопки «Применить» / «Отмена»
  - После применения — пересчёт и история

- **📤 Экспорт** — дропдаун:
  - Скачать XLSX (все колонки прайса)
  - Скачать CSV (упрощённый: SKU, название, закупка ₽, курс, цена нал, цена карта)
  - Скачать XLSX с формулами (как было в исходном Excel — для бэкапа или работы оффлайн)
  - С учётом текущих фильтров — кнопка «Экспортировать только видимые»

- **🔄 Пересчитать** — пересчитывает все товары по текущей формуле без открытия формы

## 6. ИМПОРТ XLSX/CSV — ДЕТАЛИ

### Формат файла для импорта

Поддерживаем два формата:

**1. «Простой» — для регулярных обновлений закупки:**
```
SKU         | Закупка ₽ | Курс закупа
IP17-256-B  | 64000     | 76.84
IP17-256-W  | 64000     | 76.84
...
```

**2. «Полный» — экспорт + импорт обратно:**
все колонки таблицы прайса.

### Логика парсинга

1. Определяем формат по заголовкам (есть колонка «Закупка ₽» — режим простой).
2. Идентификатор — по `SKU` (он у нас обязательный). Если кто-то решил по `ID` — селектор в модалке.
3. Парсим построчно через `xlsx` library (SheetJS). CSV — через `papaparse`.
4. Для каждой строки:
   - Находим товар → если нет, добавляем в "не найдено"
   - Если найден и значения отличаются — добавляем в "будет изменено"
   - Если найден и значения те же — в "без изменений"
5. **Показываем предпросмотр** — таблица всех изменений со старым и новым значением, цветовая подсветка
6. На «Применить» — батчевый update в БД (по 100 строк), пересчёт цен для каждого товара, запись истории с `reason='import'`
7. Финальный тост: «Импорт завершён: 423 обновлено, 5 пропущено (без изменений), 12 не найдены»
8. Скачать отчёт об импорте — кнопка после завершения

### Защита

- Лимит файла: 5 МБ
- Лимит строк: 5000
- Перед удалением / массовым изменением > 100 товаров — требуем ввести "ПОДТВЕРЖДАЮ" в модалке
- Хранить последние 10 импортов в `pricing_imports_log` для отката

Никакой логики расчёта на публичной части. Карточка товара, корзина, чекаут читают **готовые поля** из `products`:

## 13. ВАЖНЫЕ НЮАНСЫ

- **`cost_usd` — generated column**, нельзя апдейтить напрямую. Менеджер всегда меняет `cost_rub` + `cost_rate`, БД сама вычисляет.
- **Не считать цены на клиенте для отображения** — всё уже в БД. Клиент только показывает.
- **Защита от race condition** — при массовом пересчёте лочить таблицу `pricing_settings` на запись, чтобы не было двух параллельных пересчётов.
- **Цены округляются ОТ baseCash (неокруглённой)** — это правило, иначе наценки расходятся. Никаких каскадных округлений (как в исходном Excel где T = C × 1.15).
- **При увеличении курса ЦБ > 5% за сутки** — не пересчитывать автоматически, даже если `use_cbr_auto=true`. Менеджер должен подтвердить вручную. Иначе утренний скачок курса спишет всю маржу.
- **История цен** — обязательно для каждого изменения. Это аудит и возможность отката.
- **Скрытые товары (status='archived')** — в пересчёт не попадают, в прайсе по умолчанию скрыты (опция «Показать архив»).
- **Б/У товары** — у них своя логика ценообразования, не от курса. В прайсе можно показывать, но изменение формулы их не затрагивает (`type='used'` исключаем).
- **`price_override`** записывает в `override_price_cash` снимок цены на момент включения. При выключении тоггла — возвращаемся к формуле. Не теряем введённые вручную значения если потом захочется их восстановить (опционально — последнее значение хранить).
- **Минимальная маржа** — это валидация. Если после применения формулы маржа какого-то товара упала ниже `min_margin_percent`, в `notification_log` пишется запись и идёт алёрт. Цена сохраняется (формула главнее), но менеджер видит проблему.
