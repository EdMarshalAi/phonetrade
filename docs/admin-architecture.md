# Архитектура админки PhoneTrade — Промт для Claude Code

> Скопируй этот файл в проект (например, в `docs/admin-architecture.md`) и используй как спецификацию.
> Ниже описана **полная архитектура админ-панели** с подключением Supabase: все разделы, поля, связи с публичным сайтом и схема БД.

---

## 0. КОНТЕКСТ

Публичный сайт PhoneTrade уже сделан и включает:
- Главную (hero-карусель, bento-сетка категорий, новинки iPhone, trade-in блок, 3 шага обмена, 6 преимуществ, блог, полоса брендов, футер)
- Каталог категории (фильтры по модели/цвету/памяти, сортировка, карточки)
- Карточку товара (галерея, выбор памяти, цены наличные/картой, рассрочка, блоки доверия, сопутствующие)
- Корзину и чекаут (физ/юр лицо, СБП/карта/при получении/кредит, самовывоз/курьер, промокод, скидка за наличные)

**Задача админки:** дать менеджеру возможность управлять всем этим контентом, видеть заказы и аналитику, не трогая код.

---

## 1. СТЕК АДМИНКИ

- Next.js 16 App Router, route group `(admin)` → `/admin/*`
- Middleware-гард на проверку сессии (Supabase auth)
- Supabase self-hosted: `@supabase/ssr` (cookie-based auth) + service-role для server actions
- Tailwind v4, shadcn base-nova (та же дизайн-система что и сайт, но плотнее: таблицы, формы)
- react-hook-form + Zod для всех форм
- Recharts 3 для графиков
- Sonner (toast) — уведомления об успехе/ошибках
- Supabase Storage — все изображения (товары, баннеры, посты блога, лого брендов)
- Telegram Bot API + Nodemailer SMTP — уведомления о новых заказах/заявках

## 3. ДЕТАЛЬНОЕ ОПИСАНИЕ РАЗДЕЛОВ

### 3.1. Обзор (`/admin`)

**Что показывает (виджеты дашборда):**
- 4 KPI-карточки сверху: продажи за период (₽), кол-во заказов, средний чек, конверсия (заказы / визиты)
- Переключатель периода: сегодня / 7 дней / 30 дней / квартал / произвольно
- График продаж по дням (Recharts AreaChart, ось X — даты, Y — выручка)
- График заказов по статусам (Donut)
- Топ-5 товаров за период (таблица: товар, продано штук, выручка)
- Топ-5 категорий
- Последние 10 заказов (мини-таблица: №, клиент, сумма, статус, время) → клик ведёт на карточку заказа
- Лента уведомлений: новые заявки trade-in, новые заказы, низкие остатки

### 3.2. Аналитика

Раздел разбит на два независимых дашборда — поведение пользователей на сайте и финансовые показатели заказов. У каждого свой URL, свой набор фильтров и графиков.

---

#### 3.2.1. Аналитика сайта (`/admin/analytics/site`)

Назначение: понимать **что пользователи делают** на сайте, откуда приходят, где отваливаются, какие страницы и товары работают, какие — нет.

**Верхняя панель — 6 KPI-плиток** (с дельтой к предыдущему периоду в %):
1. Уникальные посетители
2. Просмотры страниц
3. Заявки (все формы: trade-in, обратный звонок, вопросы)
4. Конверсия в заказ (заказы / уник.посетители × 100%)
5. Средняя глубина просмотра (страниц за сессию)
6. Bounce rate (% сессий с одним просмотром)

**Глобальный фильтр периода:** сегодня / 7 / 30 / 90 дней / произвольно + сравнение с прошлым периодом.

**Графики и виджеты ниже:**

**А. Конверсионная воронка (главное)**
Большой sankey/funnel-чарт:
```
Визит → Просмотр карточки товара → Добавление в корзину → Начало оформления → Оформлен заказ → Оплачен
```
На каждом шаге: абсолютное число + % от шага выше + drop-off (%). Подсвечивать самый "проседающий" шаг.

**Б. Источники трафика**
Donut + таблица:
- Прямые заходы
- Поиск (organic) — с разбивкой Яндекс/Google
- Реферры (внешние сайты)
- Соцсети (VK, Telegram)
- Реклама (UTM `utm_source`)
- E-mail рассылки

Для каждого источника: визиты, конверсия в заказ, выручка с источника, ROI (если есть данные о затратах).

**В. Топ-страницы**
Таблица сортируемая:
- URL / название страницы
- Просмотры
- Уникальные посетители
- Среднее время на странице
- Bounce rate страницы
- % переходов в корзину со страницы (для карточек товаров)

**Г. Топ просматриваемых товаров vs топ покупаемых**
Две колонки рядом:
- Слева: товары с самыми высокими просмотрами карточки
- Справа: товары с самыми высокими продажами
- **Подсветить расхождение** — товары которые много смотрят, но мало покупают (проблема с ценой/наличием/описанием)

**Д. Поисковые запросы внутри сайта**
Что пользователи искали в строке поиска на сайте:
- Топ-запросы с результатами
- **Запросы без результатов** (отдельная вкладка — критично, это сигнал чего нет в каталоге)
- Запросы с низкой кликабельностью (показали результаты, но никуда не кликнули)

**Е. География посетителей**
Тепловая карта России + таблица городов: визиты, заказы, конверсия по городу. Понять реальное покрытие.

**Ж. Устройства и браузеры**
Donut по типам устройств (desktop / mobile / tablet) + браузерам. Полезно для приоритезации тестирования.

**З. Heatmap времени активности**
Сетка день недели × час суток. Видно когда пик трафика — для планирования рекламы и работы менеджеров.

**И. Hero-баннер: CTR по слайдам**
Таблица: какой слайд карусели сколько раз показан, сколько кликов, CTR%. Понять какие промо работают.

**К. Эффективность блог-постов**
Список постов: просмотры, среднее время чтения, конверсия в просмотр карточки товара со страницы поста.

**Л. Брошенные корзины**
- Сколько корзин начали оформляться и не завершили
- Средняя сумма брошенной корзины
- Какие товары чаще всего "бросают"
- Опция отправить напоминание клиенту (если оставил телефон/email)

**М. Возврат пользователей**
- Новые vs возвращающиеся (за период)
- Дней до повторного визита (медиана)
- Дней от первого визита до первой покупки (для понимания цикла принятия решения)

**Н. Core Web Vitals**
Реальные показатели скорости сайта от посетителей:
- LCP (Largest Contentful Paint)
- INP (Interaction to Next Paint)
- CLS (Cumulative Layout Shift)
- Разбивка mobile/desktop. Подсветить страницы, проваливающие пороги Google.

**О. Wishlist / избранное**
Что чаще всего добавляют в избранное (lucide-сердечко на карточках). Сигнал к будущим акциям и допродажам.

**Экспорт:** любой блок → CSV/PNG.

#### 3.2.2. Аналитика заказов (`/admin/analytics/orders`)

Назначение: понимать **сколько и на чём зарабатывает магазин**, какой ассортимент тянет выручку, какой способ оплаты выгоднее.

**Верхняя панель — 6 KPI-плиток:**
1. Выручка (₽) — оплаченные заказы
2. Кол-во заказов
3. Средний чек (AOV)
4. Кол-во проданных штук
5. Возвраты (₽ и %)
6. Маржа (если ведём учёт закупа) или Выручка наличными vs картой

**Глобальный фильтр периода + опционально фильтр по статусу заказа** (по умолчанию: оплаченные/доставленные).

**Графики и виджеты:**

**А. Динамика выручки**
AreaChart по дням/неделям/месяцам с двумя линиями: текущий период vs прошлый. Подсветка пиков.

**Б. Структура выручки по категориям**
Stacked bar или donut: iPhone / iPad / Mac / Watch / AirPods / Аксессуары / Б/У — доли в выручке и в штуках.

**В. Топ-товаров по выручке**
Таблица:
- Товар, фото
- Продано штук
- Средняя цена продажи (важно, если разные варианты памяти)
- Выручка
- Доля в общей выручке (%)

**Г. Топ-вариантов**
Какая память и какие цвета продаются лучше — для закупа. Топ комбинаций "iPhone 17 Pro 256GB Black" итд.

**Д. Способы оплаты**
Donut: СБП / Карта / При получении / Кредит / Рассрочка. Сумма и кол-во по каждому.
**Дополнительно:** доля наличного расчёта (важно, потому что есть скидка за наличные — нужно видеть реальную потерянную маржу из-за неё).

**Е. Способы получения**
Самовывоз vs курьер: кол-во, средний чек по типу.

**Ж. Воронка статусов заказа**
Сколько заказов сейчас в каждом статусе (новый / подтверждён / в сборке / готов / отгружен / доставлен / отменён). Подсветить застрявшие (давно в одном статусе).

**З. Отмены и причины**
% отмен от общих заказов + распределение по причинам (нет в наличии, передумал клиент, проблема с оплатой...). Менеджер указывает причину при отмене.

**И. RFM-анализ клиентов**
Сегментация: Recency (давно ли покупал) × Frequency (как часто) × Monetary (сколько потратил). Визуализация плиткой 3×3:
- VIP (часто, недавно, много)
- В зоне риска (раньше покупал часто, давно не было)
- Новые
- Потерянные

**К. Эффективность промокодов**
Таблица: код, кол-во применений, общая скидка (₽), привлечённая выручка, ROI промо.

**Л. Сезонность**
Тепловая карта месяц × день месяца за весь период работы — видны паттерны (Чёрная пятница, перед сентябрём, праздники).

**М. Скорость обработки заказов**
Среднее время в каждом статусе: новый → подтверждён, подтверждён → готов, готов → выдан. Помогает находить узкие места.

**Н. Маржа (опционально, если будем вести закуп)**
Если в карточке товара есть `cost_price`, считать маржу по каждому товару, категории, периоду.

**О. Trade-in метрики**
Сколько заявок trade-in, сколько конвертировалось в заказ, средняя стоимость выкупа, какие модели чаще сдают.

**Экспорт:** любой блок → CSV/XLSX, бухгалтерский отчёт за период (отдельная кнопка).

### 3.3. Заказы (`/admin/orders`)

**Список заказов:**
- Колонки: № заказа, дата, клиент (имя/телефон), сумма, способ оплаты, способ получения, статус, действия
- Фильтры: статус, период, способ оплаты, способ получения, поиск по № или телефону
- Сортировка по дате/сумме
- Bulk-actions: смена статуса, экспорт выбранных

**Статусы заказа (state machine):**
```
новый → подтверждён → в сборке → готов к выдаче → выдан (для самовывоза)
                                ↘ передан в доставку → доставлен (для курьера)
                                ↘ отменён
```

**Карточка заказа (drawer или /admin/orders/[id]):**
- Шапка: № заказа, дата создания, текущий статус (с кнопками смены), сумма
- Блок "Клиент": тип (физ/юр), имя, телефон, email, ссылка на профиль клиента
- Блок "Состав заказа": список товаров (фото, название, кол-во, цена, итого), кнопка "+ Добавить товар" (на случай корректировки)
- Блок "Получение": тип (самовывоз/курьер), адрес, желаемое время
- Блок "Оплата": способ, статус оплаты (ожидание/оплачен/возврат), сумма скидки за наличные, промокод
- Блок "Итого": товары, скидка, доставка, итого
- Блок "История": журнал действий (изменение статуса, кто, когда, комментарий)
- Блок "Комментарии менеджера" (внутренние заметки)
- Кнопки: Распечатать счёт, Распечатать чек, Отправить уведомление клиенту, Отменить, Возврат

**Создание заказа вручную** (для звонков):
- Кнопка "+ Новый заказ" наверху списка
- Форма: телефон → поиск клиента или новый, состав корзины (поиск товаров с автодополнением), способ оплаты/получения

**Уведомления:**
- При создании заказа клиентом — автоматически в Telegram чат менеджеров + email
- При смене статуса — опционально клиенту (SMS/email)

### 3.4. Заявки (`/admin/leads`)

Все нецелевые обращения с сайта:
- **Trade-in заявки**: модель, состояние (заполненное клиентом), контакты, оценочная сумма, статус (новая/связались/оценка/принято/отказ)
- **Обратный звонок**: телефон, время для связи
- **Вопрос через FAQ** (если будет форма)
- **Заявка на ремонт** (если будет сервисная форма)

Карточка заявки: контакты, история коммуникации, конвертация в заказ (кнопка "Создать заказ из заявки").

### 3.5. Товары (`/admin/catalog/products`)

**Список товаров:**
- Колонки: фото-миниатюра, название, артикул, категория, цена наличные, цена картой, остаток, статус, бейджи
- Фильтры: категория, статус (опубликован/черновик/архив), наличие, бейджи, тип (новый/Б/У), поиск
- Bulk-actions: смена статуса, смена категории, изменение цен на %, экспорт

**Карточка товара (форма создания/редактирования):**

Разбита на вкладки:

**Вкладка "Основное":**
- Название (`iPhone 17 Pro 256GB Orange`)
- Slug (auto из названия, редактируется)
- Артикул / SKU (`IP17PRO-256-ORANGE`)
- Категория (select из дерева)
- Тип товара: Новый / Б/У (toggle)
- Бейджи (multi-select): Новинка, Без RuStore, Хит, Распродажа, Эксклюзив
- Краткое описание (1-2 строки, показывается в карточке списка)
- Полное описание (rich text editor — TipTap или плейн markdown)

**Вкладка "Цены и наличие":**
- Цена наличными `price_cash` (красная цена на сайте)
- Цена картой/безналом `price_card`
- Старая цена `price_old` (для зачёркивания, если идёт акция)
- Цена в рассрочку от `price_installment_from` (отображается "От 3500 ₽/мес")
- Партнёр рассрочки (Т-Банк / другой)
- Остаток на складе `stock` (число или "уточняйте")
- Минимальный остаток для алёрта `min_stock`
- Гарантия в месяцах `warranty_months` (12+12 у магазина = 12 + 12)
- Чекбокс "Доступен для заказа"

**Вкладка "Варианты" (для товаров с памятью/цветом):**
- Таблица вариантов: каждая строка = комбинация (Память + Цвет)
  - Память (128/256/512/1TB), Цвет (Black/Silver/Sage/Orange/Cloud White...), SKU варианта, цена нал, цена карта, остаток, фото варианта
- Кнопка "+ Добавить вариант"
- Если вариантов нет — товар продаётся как есть

**Вкладка "Галерея":**
- Drag-and-drop загрузка изображений в Supabase Storage (bucket `product-images`)
- Сортировка перетаскиванием
- Главное фото (первое)
- Опционально: фото для конкретного цвета/варианта
- Alt-текст для каждого фото

**Вкладка "Характеристики":**
- Таблица key-value (например: "Чипсет: A19 Pro", "Экран: 6.3 OLED", "Камера: 48 Мп тройная")
- Группировка (Дисплей / Процессор / Камера / Батарея / Связь)

**Вкладка "Состояние" (только для Б/У):**
- Текстовое описание состояния (например, "Ни разу не разбирался, полностью в оригинале, все функции работают")
- Аккумулятор `battery_health` (число 0-100)
- Категория состояния: Идеальное / Хорошее / Удовлетворительное (для фильтра)
- Фото "как есть" (отдельная галерея в дополнение к основной)

**Вкладка "SEO":**
- Meta title
- Meta description

### 3.17. База клиентов (`/admin/customers`)

**Отдельный раздел верхнего уровня в сайдбаре** (не вложен в settings) — это рабочий инструмент менеджера, доступ нужен быстрый.

**Список клиентов:**
- Колонки: имя, телефон (главный идентификатор), email, кол-во заказов, общая сумма, средний чек, последний заказ (дата), сегмент (бейдж: Новый / Постоянный / VIP / В зоне риска), источник первого визита (UTM)
- Поиск по телефону / имени / email
- Фильтры: сегмент, период первого/последнего заказа, минимальная сумма покупок, город, источник, наличие открытых заказов
- Сортировка по любой колонке
- Bulk-actions: экспорт выбранных в CSV, добавление в сегмент для рассылки, начисление бонусов

**Карточка клиента (drawer или `/admin/customers/[id]`):**
- Шапка: имя, телефон, email, тип (физ/юр), сегмент, дата регистрации
- Блок "Статистика": всего заказов, общая сумма, средний чек, любимая категория, последний контакт
- Блок "История заказов": таблица всех заказов клиента (№, дата, сумма, статус) — клик → карточка заказа
- Блок "Заявки клиента": trade-in заявки, обратные звонки от этого клиента
- Блок "Бонусы": накопленный баланс, история начислений/списаний, ручное начисление
- Блок "Заметки менеджера": внутренние заметки (характер клиента, особенности, преференции)
- Блок "Юр.лицо" (если применимо): ИНН, КПП, название организации, юр.адрес
- Кнопки в шапке: "Создать заказ для клиента", "Позвонить" (tel: link), "Написать в WhatsApp/Telegram"

**Сегменты (правила автоматической классификации):**
- **Новый** — 0 заказов или 1 заказ < 30 дней назад
- **Постоянный** — 2+ заказа, последний < 180 дней назад
- **VIP** — сумма покупок > X ₽ (настраивается) ИЛИ 5+ заказов
- **В зоне риска** — был постоянным, но > 180 дней без покупок
- **Потерянный** — > 365 дней без активности

Сегменты считаются ночным cron-заданием + обновляются при создании нового заказа.

**Программа лояльности:**
- Накопительная скидка (правила настраиваются в `/admin/promotions/discounts`)
- Бонусы — начисление % с заказа, списание на следующий заказ
- Реферальная программа (опционально, на будущее)

**Вкладка 1 — Telegram-бот (главная)**

Все исходящие уведомления о работе магазина (новые заказы, заявки, изменения статуса) идут в Telegram. Это основной канал оперативной коммуникации с менеджерами.

**Подключение бота:**
- Поле "Bot Token" — токен бота из @BotFather (зашифрованно в БД)
- Кнопка "Проверить подключение" — дёргает `getMe` Telegram API, показывает имя бота и юзернейм, тост "Бот подключён ✅" / "Ошибка"
- Webhook URL (read-only) — куда Telegram шлёт апдейты, нужно при использовании interactive-кнопок
- Кнопка "Установить webhook" — регистрирует URL у Telegram, ставит секрет, проверяет

**Получатели (чаты):**
Список чатов куда бот шлёт сообщения. Чат может быть личный (DM с менеджером) или групповой.
- Колонки: название, chat_id, тип (personal/group), какие триггеры активны (бейджи), статус
- Добавление чата:
  - Менеджер пишет команду `/start` боту (или добавляет бота в группу и пишет `/register`)
  - Бот отправляет в чат код подтверждения
  - Менеджер вводит код в админке → чат добавляется в список
  - ИЛИ ручное добавление chat_id (для опытных)
- Для каждого чата: какие триггеры присылать сюда (мульти-чекбоксы)
- Кнопка "Тестовое сообщение" — отправляет в чат "Это тест от админки PhoneTrade ✅"

**Триггеры (события) и шаблоны:**

| Триггер | Когда срабатывает | Шаблон (с переменными) |
|---|---|---|
| `new_order` | Создан новый заказ на сайте | "🆕 Заказ {order_number} на {total} ₽. {items_count} товаров. Клиент: {customer_name} {customer_phone}. {payment_method}, {delivery_method}." |
| `order_paid` | Прошла оплата | "💰 Заказ {order_number} оплачен ({payment_method})" |
| `order_cancelled` | Заказ отменён | "❌ Заказ {order_number} отменён. Причина: {reason}" |

| `new_lead_callback` | Заказан обратный звонок | "📞 Обратный звонок: {phone}, удобное время: {time}" |
| `low_stock` | Остаток товара ≤ min_stock | "📦 Низкий остаток: {product_title} — {stock} шт. (мин: {min_stock})" |
| `abandoned_cart` | Корзина брошена > 24ч (опционально) | "🛒 Брошенная корзина {cart_value} ₽. Клиент: {phone}" |
| `daily_report` | Ежедневно в 21:00 (cron) | Сводка за день: заказов X, выручка Y ₽, новых клиентов Z |

Для каждого триггера:
- Чекбокс "Включён"
- Редактор шаблона (с подсказками доступных переменных)
- Список чатов-получателей (можно разный для разных триггеров — например, low_stock только в чат закупа)
- Inline-кнопки в сообщении (опционально):
  - В сообщении о новом заказе — кнопка "Открыть в админке" (deep link на `/admin/orders/{id}`)
  - В заявке — "Взять в работу" (меняет статус заявки на in_progress через webhook)

**Команды бота для менеджеров:**
- `/start` — регистрация

#### Таб 2 — Приглашения (`/admin/settings/users/invitations`)

Система выдачи доступа в админку: овнер/админ вводит email и роль → новому пользователю на почту приходит письмо с magic-link на установку пароля.

**Список приглашений:**
- Колонки: email, роль, статус (Отправлено / Принято / Истекло / Отозвано), отправлено когда, принято когда, кем выдано, действия
- Фильтры по статусу

**Создание приглашения (модальное окно):**
- Поле "Email" (валидация формата + проверка что email ещё не зарегистрирован)
- Поле "Имя" (опционально, чтобы письмо было персональным)
- Selector "Роль"
- Сообщение от пригласившего (опционально, добавится в письмо)
- Кнопка "Отправить приглашение"

**Что происходит после нажатия:**
1. В таблице `admin_invitations` создаётся запись: `email`, `role`, `token` (одноразовый UUID), `expires_at` (по умолчанию +7 дней), `invited_by`, `status='sent'`
2. На указанный email уходит письмо: "Вас пригласили в админку PhoneTrade. Перейдите по ссылке чтобы создать пароль: `https://phonetrade.ru/admin/accept-invite?token={token}`. Ссылка действует 7 дней."
3. В Telegram-канал админов уведомление (если включено): "Приглашение отправлено: {email}, роль {role}, от {invited_by}"

**Что происходит когда приглашённый кликает ссылку:**
1. Открывается страница `/admin/accept-invite?token={token}`
2. Проверка токена: существует, не использован, не истёк
3. Форма "Создайте пароль": email (read-only, прочитан из токена), пароль, повтор пароля
4. На submit:
   - Создаётся пользователь в Supabase Auth с указанным паролем
   - Запись в `admin_users` (id, email, role из приглашения, full_name)
   - Запись в `admin_invitations`: `status='accepted'`, `accepted_at=now()`, `user_id` ссылка
   - Аудит-запись
   - Авто-логин и редирект в `/admin`
5. Тост: "Добро пожаловать! 🎉"

**Действия с приглашением:**
- "Отправить повторно" — генерирует новый токен, обновляет письмо
- "Отозвать" — `status='revoked'`, ссылка перестаёт работать
- "Скопировать ссылку" — на случай если письмо не дошло, менеджер может передать ссылку лично

**Безопасность:**
- Токен одноразовый, после `accepted` становится невалидным
- TTL 7 дней (настраивается)
- Rate-limit на создание приглашений: 10 в час с одного админа
- Логирование всех действий с приглашениями в audit log

#### Таб 3 — Журнал действий (`/admin/settings/users/audit-log`)

Полный лог всех действий админов в системе. Закрывает аудиторский вопрос "кто что менял когда".

**Что логируется:**
- Все CRUD-операции над сущностями: товары, категории, заказы, клиенты, посты, страницы, hero, bento, настройки и т.д.
- Изменения статусов заказов и заявок
- Логины и выходы пользователей (включая неудачные попытки входа)
- Отправка приглашений и их принятие
- Изменение ролей пользователей
- Включение/отключение интеграций
- Массовые операции (bulk update)
- Импорт/экспорт данных

**Структура записи лога:**
- `id` (bigserial)
- `created_at`
- `user_id` (кто сделал; null для системных действий)
- `user_email` (снэпшот, чтобы данные оставались если юзера удалят)
- `action` — `create | update | delete | status_change | login | login_failed | logout | invite_sent | invite_accepted | role_changed | bulk_update | export | import | settings_change`
- `entity_type` — `product | order | customer | category | hero_slide | bento_tile | blog_post | page | promo_code | user | invitation | settings | integration | ...`
- `entity_id` (uuid сущности или null)
- `entity_title` (снэпшот названия)
- `changes` (jsonb с diff: `{field: {old, new}}`)
- `ip_address`
- `user_agent`

**UI лога:**
- Таблица последних действий, новейшие сверху
- Колонки: время (относительное "5 мин назад" + hover полная дата), пользователь, действие (с цветной иконкой), сущность (с ссылкой если ещё существует), кратко "что изменилось"
- Клик на строку → drawer с полным diff (старое значение → новое значение, для всех изменённых полей)
- Фильтры:
  - По пользователю (мульти-селект)
  - По типу действия (мульти-селект)
  - По типу сущности
  - По конкретной сущности (например, история одного товара или заказа)
  - По периоду
- Поиск по тексту в `entity_title`
- Экспорт результата фильтрации в CSV

**Хранение и архивирование (важно):**

Чтобы не раздувать рабочую таблицу:

1. **Горячая таблица** `admin_audit_log` — хранит последние **~200 записей** (точнее: ограничение по дням, скажем последние 90 дней, но не более 5000 строк — что наступит раньше).
2. **Архивная таблица** `admin_audit_log_archive` — то же самое схематически, без RLS-чтения из админки, только для аудита.
3. **Cron-задание раз в сутки** (Supabase Edge Function или внешний триггер):
   - Селектит записи из `admin_audit_log` старше 90 дней
   - Вставляет их в `admin_audit_log_archive`
   - Удаляет из `admin_audit_log`
4. В UI журнала по умолчанию показываются только данные из горячей таблицы.
5. Чекбокс **"Искать в архиве"** — переключает запрос на UNION с архивной (медленнее, но даёт полную историю). Доступно только роли Owner/Admin.
6. Архив не удаляется автоматически. По требованию (раз в год) можно вручную дампить старше N лет в файл и чистить.

**Пагинация:**
- 50 записей на страницу
- Сервер-сайд пагинация через Supabase `range()`
- Кнопки "Назад / Вперёд" + "Перейти к странице"

**Что НЕ логируется (чтобы не зашумлять):**
- Просмотры страниц админки (read-операции)
- Открытие drawer-ов
- Hover/scroll и прочее UI
- Внутренние системные операции (revalidateTag, cron-кроны)

**Retention policy в шапке журнала:** "Показаны действия за последние 90 дней. Архив старше — переключите фильтр «Искать в архиве»."

### 6.1. Toast-уведомления (обязательный паттерн)

**Используем Sonner** (уже в стеке). Тостер монтируется в `(admin)/layout.tsx`, расположение — нижний правый угол.

**Каждое действие админа даёт обратную связь через тост:**

| Действие | Тип | Текст |
|---|---|---|
| Создание сущности | success | "Товар «iPhone 17 Pro 256GB» создан" |
| Сохранение изменений | success | "Изменения сохранены" |
| Удаление | success | "Категория удалена" |
| Массовая операция | success | "Обновлено 12 товаров" |
| Загрузка файла | success | "Изображение загружено" |
| Отправка приглашения | success | "Приглашение отправлено на user@example.com" |
| Смена статуса заказа | success | "Заказ #PT-2026-0042 переведён в «Готов к выдаче»" |
| Тест Telegram-бота | success | "Тестовое сообщение отправлено ✅" |
| Ошибка валидации формы | error | "Заполните обязательные поля" (+ скролл к первому невалидному полю) |
| Ошибка сети / сервера | error | "Не удалось сохранить. Попробуйте ещё раз" (+ кнопка "Повторить") |
| Доступ запрещён | error | "У вас нет прав на это действие" |
| Долгая операция | loading → success/error | "Импортируем 500 товаров…" → "Импорт завершён: 487 успешно, 13 с ошибками" |
| Информационное | info | "Сайт сейчас в режиме обслуживания" |

**Правила:**
- Все mutating server actions возвращают `{ success: boolean, message: string }` и фронт показывает соответствующий тост через `toast.success()` / `toast.error()`
- Для долгих операций — `toast.promise(action, { loading, success, error })`
- Длительность тостов: success 3с, error 5с (с возможностью закрыть руками), loading — до завершения
- Никаких alert/confirm браузера — для подтверждений используем модалки с теми же тостами после выбора
- Один и тот же тост не дублируется (Sonner это умеет из коробки)
- На критичных удалениях — модалка с подтверждением вводом названия сущности ("Введите название товара чтобы подтвердить удаление")

### 6.2. Прочие UX-обещания

- **Optimistic UI** на быстрых операциях (изменение статуса, переключение чекбокса) — UI обновляется сразу, при ошибке откатывается с error-тостом
- **Loading skeletons** на всех списках вместо спиннеров
- **Empty states** с иллюстрацией и CTA ("Пока нет заказов. Когда клиенты начнут оформлять — они появятся здесь")
- **Подтверждение перед уходом** со страницы с несохранёнными изменениями (`beforeunload` + интеграция с Next router)
- **Keyboard shortcuts** в часто используемых местах: `⌘+S` сохранить, `⌘+K` глобальный поиск, `Esc` закрыть drawer/modal
- **Глобальный поиск** (`⌘+K`) — ищет по товарам, заказам, клиентам, страницам

## 8. ВАЖНЫЕ НЮАНСЫ

- **Все формы — server actions** (Next 16 App Router), не client-side fetch
- **Все списки — пагинация на сервере** (Supabase `range()`), не загружать всё разом
- **Изображения — `next/image` с remotePatterns на Supabase Storage домен**
- **Цены — `numeric` в БД, форматирование через `formatPrice()` хелпер**
- **Slug — авто из title транслитерацией, но редактируемый**
- **Дата/время — храним в UTC, показываем в МСК (`Europe/Moscow`)**
- **Загрузка файлов — через signed upload URL Supabase**
- **Удаления — soft delete** (поле `deleted_at`) где это критично (товары, заказы), hard delete где не критично (медиа, лиды)
- **Версионирование контента** опционально для блога/страниц (revisions)
- **Превью изменений** перед публикацией для контентных страниц и постов
- **Тосты — обязательны на КАЖДОЕ действие** (см. раздел 6.1). Не молчать после клика. Если действие длинное — показывать loading-тост через `toast.promise()`.
- **Аудит — обязателен на КАЖДУЮ мутацию**. Helper `logAudit({ action, entity_type, entity_id, entity_title, changes })` вызывается в каждом server action ДО возврата ответа. Без записи в audit log — действие считается недопустимым.
- **Архивирование audit log — обязательное cron-задание**: Edge Function переносит записи старше 90 дней в `admin_audit_log_archive`. Запускается раз в сутки в 03:00 МСК. Если cron не настроен — таблица разрастётся и админка станет медленной.
- **Telegram-токен и SMTP-пароли — НИКОГДА не отдавать на фронт**, использовать только server-side через service-role клиент Supabase. В UI показывать только маскированно (`••••••••XYZ`).
- **Magic-link приглашения** имеют TTL 7 дней и одноразовы — после принятия токен становится невалидным
- **Rate-limit на приглашения**: не более 10 в час с одного админа (защита от спам-инвайтов)
- **Email-уведомления клиентам обязаны иметь unsubscribe-ссылку** (152-ФЗ, общая вежливость)
